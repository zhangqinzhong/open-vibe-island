Quick Verdict

它不是
“只靠 hooks”

已安装的 Vibe Island.app 是一个本地 control plane。hooks 只是入口层， 真正撑起体验的是常驻 app、Unix socket、helper、IDE extension、Apple Events、会话索引， 以及对部分工具的第二通信通道。

System Shape

从“事件入口”到“终端回跳”的真实链路

这版 app 的关键不是某个 hook 脚本多复杂，而是它把多个局部能力叠成了完整的交互闭环。

01
CLI hooks / plugin events Claude、Codex、Gemini、Cursor、OpenCode 各自把事件送入统一 bridge。
02
Local socket server app 常驻监听 /tmp/vibe-island.sock，负责汇总事件与同步回复。
03
Session index 把 cwd、tty、rollout path、最近消息和状态持久化下来。
04
Precise jump surfaces IDE extension 通过 terminal PID 定位，Terminal / iTerm2 通过 Apple Events 定位。
05
Richer tool-specific adapters 尤其是 Codex 和 OpenCode，明显不止普通 hooks 一条通道。

Confirmed Channels

关键通信面

下面这些不是推测，而是从 app bundle、本机配置、运行中进程、已安装插件和本地落地文件里直接观察到的能力面。

1. 常驻本地 socket server

运行中的主进程 /Applications/Vibe Island.app/Contents/MacOS/vibe-island 打开了 Unix socket，并监听 /tmp/vibe-island.sock。

lsof -nP -U | rg '/tmp/vibe-island.sock|vibe-island' process: vibe-island fd: unix socket

2. bundle 内独立 helper

app 自带 Contents/Helpers/vibe-island-bridge。用户目录里的 ~/.vibe-island/bin/vibe-island-bridge 只是 launcher，不是实际逻辑主体。

  • launcher 会查找 app 路径
  • 支持 Spotlight fallback
  • 缓存 app 路径到 .bridge-cache

3. 多工具 CLI hooks

本机已被自动写入的配置文件有 ~/.claude/settings.json、~/.codex/hooks.json、~/.gemini/settings.json、~/.cursor/hooks.json，都统一指向 ~/.vibe-island/bin/vibe-island-bridge。

  • Claude: 接了 Notification / PermissionRequest / PreToolUse / Stop / UserPromptSubmit 等完整事件。
  • Codex: 当前只接了 SessionStart / UserPromptSubmit / Stop。
  • Gemini / Cursor: 也都已经接入。

4. Claude statusline 旁路

Claude 还额外挂了 statusLine.command，把 .rate_limits 写到 /tmp/vibe-island-rl.json。这说明 quota / usage 展示不是普通 hooks 自己能给出来的。

~/.vibe-island/bin/vibe-island-statusline /tmp/vibe-island-rl.json

5. IDE terminal focus extension

本机同时安装了 Cursor 和 VS Code 扩展 vibe-island.terminal-focus。扩展通过 registerUriHandler 接 URI，然后按 terminal shell PID 扫描 vscode.window.terminals，定位并聚焦正确 tab。

纯 hooks 做不到

shell hook 最多告诉你当前进程和 cwd，没法直接把 Cursor / VS Code 里“正确的 integrated terminal tab”聚焦出来。

  • 没有 IDE 内部 terminal 列表访问权
  • 没有 PID → terminal tab 的映射面
  • 没有 IDE 内部 focus API

Vibe Island 的做法

安装 IDE extension，把 “终端 tab 聚焦” 变成 IDE 内部动作。主 app 只要把 PID 传进去，扩展就能在 IDE 进程里完成最后一跳。

  • URI handler 负责接命令
  • extension 拿到 terminal.processId
  • 匹配后执行 terminal.show(false)

6. Apple Events / AppleScript

bundle 申请了 Apple Events 权限，二进制里直接内嵌了针对 iTerm2 和 Terminal 的 AppleScript 片段。

  • com.apple.security.automation.apple-events
  • tell application "iTerm2"
  • tell application "Terminal"
Persistence Layer

会话索引不是临时 UI 幻觉

它把 session 与 terminal 的关联关系写进本地文件。这意味着 jump、摘要、注意力状态和多 session 视图都站在一层长期索引之上。

session-terminals.json

本机落地文件位于 ~/Library/Application Support/vibe-island/session-terminals.json。里面不是只有 session id，而是整层 terminal / assistant / rollout 元数据。

cwd tty currentTool status codexRolloutPath lastAssistantMessage

这层索引解决了什么

Jump-back 不是盲跳 有 tty、tool type、工作目录、rollout path，才能决定该激活哪个终端、哪个会话、是否去 IDE。
摘要不只靠当下事件 持久化最近一条 assistant / user message，意味着面板可以在 hook 沉默时继续显示上下文。
重启后还能恢复部分上下文 这也是为什么它看起来不像 demo，而更像一层常驻控制面。
Beyond Hooks

Codex 与 OpenCode 的第二通道

这里是最值得后续做动态拆解的部分。静态证据已经足够说明：至少对部分工具，它不是只等 hook 事件推送。

OpenCode: in-process reply loop

本机存在 ~/.config/opencode/plugins/vibe-island.js。这个插件会直接连 Unix socket，同时还会对本地 localhost 接口回写 permission / question reply。

  • sendToSocket: fire-and-forget
  • sendAndWaitResponse: held connection 等结果
  • permission reply: http://localhost:${serverPort}/permission/{id}/reply
  • question reply: http://localhost:${serverPort}/question/{id}/reply
  • Ghostty: 还会写 OSC 2 tab title

Codex: 明显存在 app-server / watcher 通道

虽然本机 ~/.codex/hooks.json 只接了很少的事件，但主 binary 里出现了完整的 Codex app-server / WebSocket / desktop approval watcher 痕迹。

CodexAppServerClient CodexAppServerManager CodexDesktopApprovalWatcher webSocketTask ws://127.0.0.1:0 http://127.0.0.1: .codex/session_index.jsonl ~/.codex/sessions

这几乎可以直接判定：Codex 在安装版里不止 hooks 一条线，后续动态拆解就该追这条本地 app-server / file watcher 通道。

Installed Surfaces

它到底装了哪些东西

把它看成一组“协作模块”更准确，而不是单个 app。真正的价值在于这些模块彼此补位。

Main App

  • /Applications/Vibe Island.app
  • menu bar app
  • socket server
  • UI / analytics / updates

Bridge Layer

  • Contents/Helpers/vibe-island-bridge
  • ~/.vibe-island/bin/vibe-island-bridge
  • multi-source adapter entrypoint

Tool Surfaces

  • Claude hooks + statusline
  • Codex hooks + likely app-server
  • Gemini hooks
  • Cursor hooks
  • OpenCode plugin

Terminal Jump Surfaces

这也是安装版最有产品壁垒的一块：它不是只会“弹通知”，而是会尽可能把你送回对的地方。

Terminal / iTerm2 通过 Apple Events / AppleScript，加上 TTY / pane / tab 线索来做原生终端聚焦。
Cursor / VS Code 通过已安装 extension 的 URI handler，加 PID 匹配 terminal.processId 来做 IDE 内部终端聚焦。
Ghostty / other shells 还在尝试用 terminal title / OSC 2 /环境变量补足线索。

External Services

  • Sparkle
  • Sentry
  • PostHog
  • Anthropic OAuth usage
  • sound pack registry
  • LemonSqueezy
Bottom Line

为什么这份拆解重要

如果我们后面要把 OSS 版做强，重点不是“把 hooks 写得更花”，而是把 control plane 补齐。

普通 hooks 能做的

  • 收到一些离散事件
  • 拿到部分 cwd / tool / prompt 线索
  • 在支持同步返回的工具里做基础 approval gating

普通 hooks 做不到的核心体验

  • 精准聚焦 IDE 内正确 terminal tab
  • 跨工具统一的 session 索引与恢复
  • Claude quota 这类 statusline 旁路数据
  • OpenCode 这种 in-process reply loop
  • Codex 这种明显存在第二通道的 richer integration
